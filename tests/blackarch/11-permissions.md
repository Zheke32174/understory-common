# 11 — Permission audit

**Threat class**: supply-chain risk via transitive deps adding
permissions silently.

**passgen defense**: every comms / sensor / storage / install / overlay
/ sip / a11y-bind / device-admin / location / contacts / calendar /
audio / camera / biometric-legacy permission is explicitly stripped
in `AndroidManifest.xml` via `tools:node="remove"`. If a future
dependency tries to add one, the merged manifest still removes it.

## Test 1: badging dump

```bash
APK=android/dist/passgen.apk
$ANDROID_HOME/build-tools/35.0.0/aapt2 dump badging "$APK" | grep ^uses-permission
```

**Expected** (the only line):

```
uses-permission: name='android.permission.USE_BIOMETRIC'
uses-permission: name='com.understory.passgen.DYNAMIC_RECEIVER_NOT_EXPORTED_PERMISSION'
```

`USE_BIOMETRIC` is required for `BiometricPrompt` (the
device-credential gate on the vault) — accepted as the one user-visible
permission.

`DYNAMIC_RECEIVER_NOT_EXPORTED_PERMISSION` is auto-generated by AGP for
internal `Context.registerReceiver` security; it has signature-level
protection and does NOT appear in the user-facing install dialog.

If you see anything else (INTERNET, ACCESS_NETWORK_STATE, READ_PHONE_STATE,
SEND_SMS, etc.), a transitive lib has snuck a permission in and the
strip list needs updating.

## Test 2: install-dialog presentation

```bash
adb install android/dist/passgen.apk
```

The system installer prompt should show only the biometric permission
(if it shows any — most modern Android installers don't show biometric
either, since it's signature-classified for the user's authenticator).

## Pass/fail

✅ Test 1: only USE_BIOMETRIC + DYNAMIC_RECEIVER permission present
✅ Test 2: install-dialog shows minimal or zero user-visible permissions
