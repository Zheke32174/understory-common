# 07 — Malicious accessibility service

**Threat class**: privileged screen reader / event injector abuse
**Tool**: any third-party app declaring `BIND_ACCESSIBILITY_SERVICE` and
asking the user to enable it. Once enabled, an a11y service can read
text content of every app's UI tree and inject taps/typing.

**passgen defense**: structural — passgen never renders password values
to a `TextView` (master is base64'd in entry[0] but only revealed via
biometric-gated 10s window; saved entries' passwords never reach
on-screen text in any view we control). Delivery is via:
- `commitText` IPC (IME mode) — bypasses the source-side a11y tree
- Autofill IPC (autofill mode) — bypasses the source-side a11y tree
- Clipboard with EXTRA_IS_SENSITIVE — Gboard skips, Samsung known leak

Plus `A11yProbe` surfaces a banner when any third-party a11y service is
enabled, so the user knows the threat surface.

## Setup

Build a minimal test accessibility service:

```kotlin
class TestA11y : AccessibilityService() {
    override fun onAccessibilityEvent(event: AccessibilityEvent) {
        Log.d("test-a11y", "event from ${event.packageName}: " +
            "type=${event.eventType} text=${event.text}")
    }
    override fun onInterrupt() {}
}
```

Manifest:
```xml
<service android:name=".TestA11y"
    android:permission="android.permission.BIND_ACCESSIBILITY_SERVICE">
    <intent-filter>
        <action android:name="android.accessibilityservice.AccessibilityService" />
    </intent-filter>
    <meta-data android:name="android.accessibilityservice"
        android:resource="@xml/a11y_config" />
</service>
```

`a11y_config.xml`:
```xml
<accessibility-service
    android:accessibilityEventTypes="typeAllMask"
    android:accessibilityFeedbackType="feedbackGeneric"
    android:notificationTimeout="0"
    android:canRetrieveWindowContent="true" />
```

Install, then enable in Settings → Accessibility → Installed apps.

## Test 1: A11yProbe detects and warns

After enabling the test a11y service, open passgen.

**Expected**: a banner appears at the top of MainActivity:
"⚠ 1 third-party accessibility service(s) active. Accessibility services
can read text on screen and inject taps. Tap to review."

If the banner doesn't appear, A11yProbe failed to detect — investigate.

## Test 2: a11y service can't read passwords

With the test a11y service running and logging events:

```bash
adb logcat | grep test-a11y
```

Open passgen, set up vault if needed, view the master entry, do "Show
password" with biometric auth.

**Expected**: a11y event logs show events for buttons, labels, the
"●●●●●●●" placeholder text. They do NOT show the actual password
characters when reveal is active — even though the password is on
screen for those 10s, it's rendered via Compose's `Text` composable
which marks itself with appropriate semantics.

**Honest scope**: when the password IS revealed (during the 10s
window), Compose Text composables are accessible to screen readers by
default. If you want stricter, we'd need to mark those Text composables
with `Modifier.semantics { invisibleToUser() }`, which would also
break legitimate TalkBack usage. Trade-off; current default is "visible
to a11y when revealed."

## Test 3: a11y can't observe IME-typed values

Use the passgen IME on a password field in another app, with the test
a11y service active and logging.

**Expected**: a11y logs show focus events on the target app's password
field, "type" events as text appears, but the text content reported is
the *masked* representation (dots/asterisks) on a `inputType=textPassword`
field, not the actual characters. This is Android's standard behavior
for password inputs — we don't need a defense, the platform handles it.

If the target app uses `inputType=text` (visible password mode), the a11y
service WILL see the actual chars. Same as any IME; not specific to
passgen.

## Pass/fail

✅ Test 1: A11yProbe banner appears when test service is enabled
✅ Test 2: a11y events don't carry password chars when reveal is closed
⚠ Test 3: depends on target app's input type. Documented limitation.
